# 📰 デイリー・ニュース記事 手順書（NEWS DAILY SOP）

毎日夕方に「その日いちばん話題性が高く、マーケット影響が大きいニュース」を**1本だけ**記事化する予約エージェント（routine）の作業手順書です。
**公開はしません。下書きを `drafts/news/` に置き、台帳に判定を記録するだけ**。公開は人間が下書きを一目見て OK したときだけ（ワンクリック公開）。

> 設計思想（最重要）：
> 1. **薄い日は無理に出さない。** 「毎日1本」より「出すなら価値がある1本」。基準に満たなければ**見送り**して台帳に理由だけ残す。
> 2. **ニュースは事実誤認・無登録投資助言・景表法の“最高リスク”領域。** 「○○を買え/売れ」は絶対に書かない。「何が起きたか・なぜ・どう影響しうるか」の**中立整理**に徹する。
> 3. **routine は本番ファイル（guides.html / generate_market_news.py / index 等）に絶対触れない。** 触るのは `drafts/news/` のみ。公開は人間＋（必要なら）Opus コンプラ監査を経てから。

---

## 毎回の作業手順

### 1. 文脈を読む
- `CLAUDE.md`（サイト全体像・コンプラ・8ステップ・SYNC禁忌）。
- **テンプレ基準＝直近の速報系記事**（例 `guide-btc-crash-2026-06.html` や `guide-bank-stocks-2026-05.html`）を Read。デザイン・3段階整理（🟢🟡🔴）・出典明記・kinsho-v1免責の手本。
- **重複回避のための既掲チェック**：`drafts/news/NEWS_LEDGER.md`（過去の選定・見送り履歴）／`guides.html`（最近の速報カード）／`political-feed.json`（政治発言フィードで既に扱った話題）／直近の `guide-weekly-*` `guide-*-2026-*`。**すでに当サイトで十分扱った話題は選ばない。**

### 2. ニュースを集める（WebSearch）
その日（JST基準の当日〜前夜）の市場ニュースを複数の角度で検索して候補を 5〜10 本リストアップ：
- マクロ・政策（FOMC/日銀/CPI/雇用統計/GDP/要人発言/関税・規制）
- 指数・為替・コモディティ（日経・S&P・ドル円・金・原油・米金利の大きな動きとその理由）
- 地政学・市況イベント
- **個別銘柄ニュース**（決算・新製品・M&A・不祥事・急騰急落）※対象に含む。ただし扱いは後述の「個別銘柄ルール」厳守。
- 各候補に「何が起きたか」「一次に近い出典」をメモ。

### 3. スコアリングして1本に絞る
各候補を 0〜5 で採点（合計15点満点）：
| 軸 | 0〜5 の意味 |
|---|---|
| **話題性** | その日どれだけ報じられ・検索され・関心を集めているか |
| **マーケット影響** | 日経/S&P/為替/金利/コモディティ/主要銘柄をどれだけ動かす・動かしうるか |
| **付加価値** | 見出し1行で終わらず、**背景・なぜ・影響の考え方・チェックポイント**を中立に1500〜2500字書けるだけの“中身”があるか |

- 最高得点の1本を選定。**同点なら「日本人投資家に身近な方」**を優先。

### 4. ★見送り判定（薄ければ出さない・ここが肝）
次のいずれかに当てはまれば**記事を作らず見送り**。`NEWS_LEDGER.md` に日付＋「見送り」＋理由＋候補一覧だけ記録して終了：
- 最高得点が **合計 9/15 未満**。
- **付加価値が 2 以下**（中立に書ける“中身”が無い＝見出し速報の焼き直しにしかならない）。
- 事実が**まだ固まっていない**（発表待ち・観測記事・確報前）。憶測で書かない。
- すでに `political-feed` / 手動速報 / 最近の guide で**十分扱った話題**。
- 中立に書くのが困難で、どう書いても投資勧誘・煽りに寄ってしまう題材。

> 「毎日出す」より「見送る勇気」。空振りの日は1行ログだけでOK。

### 5. 事実確認（公開前提の速報なので最重要）
選定したら、記事に載せる**日付・数値・固有名・発言・出典**を WebSearch で**最低2系統**照合：
- 数値（株価・指標値・前年比・規模）は確証が取れたものだけ。曖昧なら載せない or 「報道ベース」と明記。
- 「いつ起きたか」を必ず確認（CLAUDE.md：速報は日付の事実確認必須）。
- 一次情報（公式・当局・決算資料）に近い出典を優先し、本文 or 末尾に出典を明記。

### 6. 下書きHTMLを書く → `drafts/news/draft-news-<YYYY-MM-DD>.html`
直近の速報記事テンプレを**そのまま踏襲**：
- head（title/description/keywords/canonical/OGP/JSON-LD/GA/fonts）。**`<meta name="robots" content="noindex,nofollow">` を必ず入れる**（下書きは検索除外）。
- `<style>` はテンプレ記事をコピー（ライト基調・`.info-box`/`.warning-box`/`.disclaimer-banner` 等）。
- nav 9ボタン（順序厳守）・breadcrumb・**冒頭 disclaimer-banner**・h1・meta-line（カテゴリ＝「今日のニュース」）。
- 構成：**「結論3行サマリ」→ 何が起きたか → なぜ起きたか（背景）→ マーケットへの影響の考え方（🟢中立/🟡留保/🔴警戒の3視点）→ 投資家のチェックポイント → 関連記事 → kinsho-v1免責 → footer**。
- 断定を避け、複数の見方を併置。将来の値動きは予測しない（「こう動きうる/こういう論点」まで）。
- 出典は「（出典: ○○）」形式で明記。SVG概念図は任意（実価格を断定的に図にしない）。
- メルマガ登録ブロック（既存記事末尾の MailerLite 部）を踏襲。

#### 個別銘柄ニュースのルール（対象に含むが要注意）
- **「買い/売り」「狙い目」「今が好機」「一択」等は禁止。** タイトルも本文も**フラット整理**（例：「○○が△％急騰、なぜ？ 5つの要因と3シナリオを中立整理」）。
- 値上がり/値下がりの**事実と背景・リスク両面**を必ず併記。「上がる」と読ませない。
- 既存の個別銘柄解説（NVIDIA/AMD/トヨタ等）のトーンを手本に。
- 冒頭バナー＋末尾免責で「情報提供であり投資助言ではない／特定銘柄の売買推奨ではない」を明示。

### 7. SEO 仕上げ（seo-ux 役）
- title（32〜40字目安・キーワード前方）・meta description（120字前後）・OGP・JSON-LD（NewsArticle か Article）。
- 関連する既存 guide へ内部リンクを2〜3本（例：CPIニュース→`guide-us-cpi.html`、日銀→`guide-boj-*` 等）。**学習ロードマップ `guide-learning-roadmap.html` への動線も1本**。

### 8. コンプラ自己監査（必須・黒/グレー/白）
保存前に自分でチェックし、違反は直す：
- ❌ 個別銘柄の売買推奨・「買え/売れ/狙い目/一択」→ 中立整理に直す。
- ❌ 断定・利益保証（「絶対」「必ず」「100%」「確実に上がる/下がる」「儲かる」）→ 削除/軟化。
- ❌ 煽り・不安扇動（過度な恐怖訴求）→ トーンを丸める。
- ❌ 確証の無い数値・固有名・発言 → 削除 or 出典明記。
- ✅ kinsho-v1 免責（冒頭バナー＋本文末＋footer）。
- 自分で白に直せない/微妙＝**「🚩要人間レビュー（コンプラ）」**を台帳とREVIEWに明記（公開可否は人間が判断）。

### 9. 台帳に記録 → `drafts/news/NEWS_LEDGER.md`（最新を上に追記）
1エントリに：
```
## <YYYY-MM-DD>
- 判定: 公開準備OK / 🚩要人間レビュー / 見送り
- 選定トピック: <タイトル案>（カテゴリ: マクロ/個別銘柄/政策…）
- スコア: 話題性 x / 影響 y / 付加価値 z （合計 n/15）
- 出典: <URL を2本以上>
- コンプラ自己監査: 🟢白 / 🟡（直した点） / 🚩要相談
- 下書き: drafts/news/draft-news-<日付>.html
- （見送り時）理由: ___ / その日の候補上位3本: ___
```

### 10. 保存（出力先）
- **ローカルの Claude Code 予約タスク（`news-daily`）として動かす場合**：`drafts/news/draft-news-<日付>.html` と `drafts/news/NEWS_LEDGER.md` を**ローカルに保存するだけ**（GitHub へ push しない・git 操作しない）。人間がローカルでレビュー→`mw publish` で公開する。
- （将来 claude.ai のクラウド routine に移植する場合のみ：上記2ファイルを GitHub(main) にコミット。）
- **どちらの場合も `guides.html`・`generate_market_news.py`・`sync_to_github.py`・index 等の本番ファイルには絶対に触れない／コミットしない。**
- **1日1本だけ**。当日分の draft が既にあれば何もしない。

## 公開までの流れ（人間＝あなた＋メインClaudeが行う・ワンクリック）
1. 夕方以降、`drafts/news/NEWS_LEDGER.md` の当日エントリを見る（公開準備OK / 🚩要レビュー / 見送り）。
2. 「公開準備OK」なら下書きを一目チェック。気になればメインClaudeが微修正＋（個別銘柄や際どい題材は）**compliance-reviewer(Opus)監査**。
3. OKなら `guide-news-<日付>-<スラッグ>.html` として確定し、`python mw.py publish --file ... --category "今日のニュース" --emoji 📰 --card-title ... --desc ...` → sync → workflow → ライブ確認。
4. 「見送り」の日は**何もしない**（それで正常）。
- ＝「公開されるのは必ず人が一目通したもの」を担保。慣れて品質が安定したら自動公開への格上げを検討（signal-lab と同じ順序）。

## ファイル/カテゴリ規約
- 下書き：`drafts/news/draft-news-<YYYY-MM-DD>.html`（noindex・`/drafts/` は robots Disallow 済）。
- 台帳：`drafts/news/NEWS_LEDGER.md`。
- 公開時ファイル名：`guide-news-<YYYY-MM-DD>-<英小文字スラッグ>.html`。
- 公開カテゴリ：guides.html「🔥 速報・タイムリー記事」、バッジ「今日のニュース」。
- これらの routine 出力（`drafts/news/*`）は **SYNC外**（GitHub 側で生成。ローカルから push 禁止＝巻き戻し事故防止）。本 SOP（`drafts/NEWS_DAILY_GUIDE.md`）だけは人間が編集するので SYNC_FILES 入り。
