# 🤖 下書き自動生成ルーティン 手順書（AUTODRAFT SOP）

このファイルは、毎日1本「投資の心理・メンタル／リスク管理」シリーズの **下書き** を自動生成する
予約エージェント（routine）が読むための作業手順書です。**公開はしません。下書きを `drafts/` に置くだけ**です。

> 設計思想（重要）：このサイトの生命線は **コンプラ（無登録投資助言業・景表法を犯さない）と品質**。
> よって自動化するのは「**創作の下書き**」まで。**公開前に必ず人間＋compliance-reviewer(Opus)監査を通す**。
> routine は本番ファイル（guides.html / generate_market_news.py / index 等）には **絶対に触れない**。

---

## 毎回の作業手順

### 1. 文脈を読む
- `CLAUDE.md`（サイト全体像・コンプラ・8ステップ）と、テンプレ基準として **`guide-loss-cut.html`（公開済みの第1弾）を必ず Read**。
  これが**デザイン・構造・SVGクラス体系・免責(kinsho-v1)・二層構造の手本**。完全に踏襲する。

### 2. 次に書く topic を1つ選ぶ
下の「topic キュー」を上から見て、**まだ下書きが無く（`drafts/draft-<key>.html` が存在しない）、まだ公開もされていない（`guides.html` に `guide-<key>.html` が無い）最初の topic** を1つ選ぶ。該当が無ければ「全て下書き済み」と REVIEW ノートに書いて終了（新規生成しない）。

### 3. 事実確認（WebSearch）
- topic の核となる用語・数値・人名・出典を **WebSearch で照合**（例：プロスペクト理論、2%ルール、最大ドローダウン、複利、シャープレシオ 等）。
- 不確実な数値・固有名は載せない。載せる場合は出典を本文 or 末尾に明記。

### 4. 下書きHTMLを書く → `drafts/draft-<key>.html` に保存
`guide-loss-cut.html` の構造を**そのまま踏襲**：
- head（title/description/keywords/canonical/OGP/JSON-LD/GA/AdSense/fonts）。**ただし `<meta name="robots" content="noindex,nofollow">` を必ず入れる**（下書きは検索除外）。
- `<style>` は guide-loss-cut.html をコピー（ライト/ダーク両対応・SVGクラス `.s-*` 一式）。新しい図に必要なら `.s-*` クラスを追記。
- nav 9ボタン（順序厳守）・breadcrumb・disclaimer-banner・h1・meta-line（カテゴリ＝該当シリーズ名）。
- **「この記事の結論（3行サマリ）」→ 目次(toc)→ 本文セクション→ CTA→ 関連記事→ kinsho-v1免責→ footer**。
- **二層構造**：前半セクション＝初心者向けの入口（用語・なぜ大事か）、後半セクション＝中上級向け（実践・数式・期待値・当サイトのデータ/思想との接続）。
- **SVG概念図を2〜3点**入れる（実価格は使わず概念図＋キャプション＋「※概念を示すイメージ図です」）。
  - 自信が持てない図は、無理に描かず `<!-- TODO(SVG): 〇〇の概念図を要追加・要ライト/ダーク確認 -->` を残す（後で人間が仕上げる）。
- section後半に「当サイトでの活かし方」を入れ、track-record / 関連 guide へ内部リンク。CTAは track-record.html。
- 関連記事カードは loss-cut / 該当シリーズの近接topic / track-record などから3枚。

### 5. コンプラ自己チェック（必須・本文に違反があれば直してから保存）
- ❌ 個別銘柄の売買推奨（「○○を買え」「今エントリー」）→ 教育・一般論に留める。
- ❌ 断定・利益保証（「絶対」「必ず」「100%」「一択」「保証」「儲かる」）→ 使わない。むしろ損失・リスクを誠実に書く。
- ✅ kinsho-v1 免責（記事冒頭バナー＋本文末＋footer）が入っているか。
- ✅ 出典・事実が妥当か。

### 6. レビューノートを書く → `drafts/REVIEW.md`（追記・最新を上に）
1エントリに：日付 / topic / 生成ファイル名 / 参照した出典URL / 自己コンプラチェック結果 / **人間がやる残作業（SVGの実機ライト/ダーク確認・Opus compliance-reviewer監査・タイトル微調整）**。

### 7. コミット
- `drafts/draft-<key>.html` と `drafts/REVIEW.md` のみを GitHub(main) にコミット。
- **`guides.html`・`generate_market_news.py`・`sync_to_github.py`・index 等の本番ファイルには絶対に触れない／コミットしない。**
- 1日1本だけ。既に当日分の draft があるなら何もしない。

## 公開までの流れ（人間＝あなた＋メインClaudeが行う）
1. `drafts/REVIEW.md` で当日の下書きを確認。
2. メインClaudeが下書きを仕上げ（SVGの実機ライト/ダーク確認・微修正）→ **compliance-reviewer(Opus)監査**。
3. 🟢になったら `guide-<key>.html` として確定し、`mw publish`（or ドリップ用キューに投入）→ sync → workflow → ライブ確認。
4. ＝「公開されるのは必ずレビュー済み」を担保。
